# JDO 车机自动化测试平台 —— 需求文档

> 版本：v0.1（原型阶段）  
> 最后更新：2026-06-25  
> 状态：设计阶段告一段落，进入「边做边改」的开发阶段

---

## 一、项目背景

### 1.1 业务现状
JDO 是车机端（IVI）生态 App 的测试团队，被测对象为车机上的生态 App：**酷我音乐、喜马拉雅、乐听、爱奇艺、Launcher、车内会议 App** 等。这些 App **单个业务逻辑不复杂，但数量多**；且需要在**多个 OEM 品牌车机**（奥迪、保时捷、大众、宝马、奔驰……）上分别验证。

每次发版回归，测试团队需要在不同品牌车机上**手工重复**执行相同的回归用例，**人力消耗巨大**，且容易遗漏、效率低。

### 1.2 项目目标
搭建一个**车机生态 App 自动化回归测试平台**，实现：
- **一次写用例、多品牌跑、自动出报告**；
- 量化「自动化覆盖率」与「节省人力」，作为团队成果与持续投入的依据；
- 替代重复手工回归，把人力释放到探索性测试与新功能验证上。

### 1.3 技术前提（已确认）
- **系统**：Android Automotive，**全部车机可开启 ADB**（debug 包，无签名问题）；
- **硬件**：测试**台架**（裸头单元），非整车；初期 1–2 台；
- **核心约束**：同一生态 App 的 APK 在各品牌上基本一致，App 内部 UI（resource-id / text）跨品牌稳定，**品牌差异主要在 Launcher、分辨率、系统外壳** → 用例逻辑写一遍，品牌差异收敛到「设备/品牌适配层」。

### 1.4 技术栈（已确认）
| 层 | 选型 |
|---|---|
| 测试框架 | Python + pytest + uiautomator2(openatx) + adbutils(ADB Helper) + POM 分层；多设备并行 pytest-xdist |
| 平台后端 | FastAPI + SQLite（后期可换 PostgreSQL） |
| 执行引擎 | subprocess 调 pytest + `pytest-json-report` 解析结果落库 |
| 报告 | Allure + 失败现场（截图 / 录屏 / logcat） |
| 平台前端 | Vue3 + Element Plus + ECharts（原型期 CDN 免构建，正式期可上工程化） |
| 通知 | 企业微信 / 钉钉 Webhook |

## 二、核心设计思想

### 2.1 数据主线（开发依赖顺序）
```
功能用例（测什么·手工清单·覆盖率分母）
   └─关联─> 自动化脚本（怎么测·代码实现·覆盖率分子）
              └─组装─> 测试任务（App×品牌范围）
                         └─执行─> 执行中心（多品牌并行）
                                    └─产出─> 测试报告（留证+Allure）
                                               └─汇总─> 覆盖率看板
```
这条主线是平台骨架，**也是开发的推荐落地顺序**。

### 2.2 覆盖率口径（已确认）
- **主指标：业务用例覆盖率 = 已自动化测试点 ÷ 全部手工回归测试点**；
- **派生指标：预估节省人力（人天/版）** —— 领导汇报用；
- 分母来自测试团队的手工回归 checklist（Excel 导入）；
- **自动化状态由脚本标记自动判定**（脚本里 `@pytest.mark.case("KW-PLAY-001")`），不允许人工填写，保证覆盖率真实可信。

### 2.3 跨品牌策略
| 品牌差异点 | 应对 |
|---|---|
| Launcher 入口不同 | 不点图标，直接 `adb am start` 按包名/Activity 拉起 App |
| 分辨率/比例不同 | 定位优先 resource-id/text，禁止写死坐标 |
| 个别品牌定制 App | 基础 POM + 品牌差异化覆盖，只在差异处覆盖 |

### 2.4 关键校验原则
- **音乐/音频类**（酷我、喜马拉雅、乐听）：用 `adb shell dumpsys media_session` 校验真实播放状态，不只看 UI；
- **视频类**（爱奇艺）：播放面为 secure surface（HDCP），截图可能黑屏，靠 media session 状态判断；
- **失败现场三件套**：截图 + 录屏 + logcat，保证可定位。

## 三、模块需求与实现逻辑

> 平台共 **11 个功能模块 + 登录**，侧边栏按「概览 / 测试管理 / 执行与结果 / 资源 / 系统」分组。

### 3.0 登录与权限（预留）
- **需求**：登录页（账号/密码/记住我/忘记密码）；侧边栏底部用户区（头像+姓名+角色+退出）；顶部栏右上角用户头像。角色：管理员 / 测试工程师 / 只读。
- **实现逻辑**：FastAPI + JWT；角色权限控制接口（RBAC，后期细化）；前端路由守卫。原型阶段仅预留界面与入口。

### 3.1 工作台（dash）
- **需求**：进门总览。关键指标卡（业务用例覆盖率 / 今日执行 / 执行成功率 / 在线设备）+ 最近执行列表 + 快捷入口（新建任务、上传脚本）。
- **实现逻辑**：后端聚合接口返回汇总数据；最近执行取报告表 top N。

### 3.2 测试用例（cases）—— 覆盖率分母
- **需求**：
  - 手工回归用例库（功能用例：用例编号 / 所属App / 功能模块 / 优先级 / 自动化状态 / 关联脚本）；
  - 筛选：所属 App、优先级、**自动化状态**；
  - **Excel 导入**（批量灌入分母）、**Excel 导出**（评审/备份）、新增功能用例；
  - 「关联脚本」列体现每条用例由哪个脚本自动化（与脚本管理双向映射）。
- **实现逻辑**：
  - 用例表 `case(id PK=用例编号, app, module, priority, ...)`；
  - 导入：解析 .xlsx，**以「用例编号」为主键 upsert**（存在则更新、不存在则新增），返回新增/更新/错误行统计；提供导入模板下载；
  - **自动化状态不入库填写**，由脚本扫描 `@pytest.mark.case` 标记动态计算；
  - 导出：当前筛选结果 → .xlsx。

### 3.3 脚本管理（scripts）—— 覆盖率分子
- **需求**：
  - 上传 `.py / .zip` 脚本，**关联到手工用例**（核心映射，支持多选/全选）；
  - 列表展示：脚本 / 关联App / 关联用例数 / 版本 / 最近结果；
  - **点开脚本行展开** → 查看其覆盖的手工用例明细；
  - **每行下载按钮**，方便他人拉到本地执行。
- **实现逻辑**：
  - 脚本表 `script(id, name, app, version, file_path, ...)` + 映射表 `script_case(script_id, case_id)`；
  - 上传：存文件 + 入库；理想做法是**扫描脚本里的 `@pytest.mark.case` 自动建立映射**（避免人工维护脱节），原型期支持手工勾选；
  - 下载：返回脚本文件流。

### 3.4 测试任务（tasks）
- **需求**：把用例组合成回归套件，指定 App 范围 + 品牌范围 + 用例范围（全部/仅P0/仅P1）+ 执行方式（并行/串行）；列表展示任务，可直接「执行」。
- **实现逻辑**：任务表 `task(id, name, apps[], brands[], scope, mode)`；新建任务弹窗实时预估用例数（App数 × 品牌数 × 平均用例）；「执行」跳转执行中心并预填。

### 3.5 执行中心（exec）
- **需求**：选任务 + 勾选品牌 → 一键执行；**多品牌并行执行矩阵**（行=用例，列=品牌，实时刷新状态）+ 进度条 + 实时日志。
- **实现逻辑**：
  - 后台异步任务，对每个「品牌×用例」调度 pytest（按设备并行）；
  - 通过 WebSocket / SSE 推送实时状态到矩阵与日志；
  - 执行完落库为一条报告 + 明细。

### 3.6 测试报告（reports）—— 报告中心
- **需求**：
  - 工具栏筛选：**日期范围 + App + 品牌 + 状态 + 查询 + 导出 Excel**；
  - 列表：编号 / 任务 / 品牌 / 通过数 / 耗时 / 结果 / 时间 / Allure 入口；
  - **点击行展开详情**：统计 chip（通过/失败/耗时/设备/触发方式）+ 用例结果列表 + 失败用例的日志/截图/录屏 + **打开 Allure** 完整报告。
- **实现逻辑**：报告表 `report(id, task, app, brand, pass, total, dur, status, time, trigger)` + 明细表 `report_case`；Allure 报告静态目录由执行引擎生成并托管，详情页跳转。

### 3.7 覆盖率看板（cov）
- **需求**：核心指标卡（手工回归测试点 / 已自动化 / 业务用例覆盖率 / 预估节省人力）+ 覆盖率增长趋势（按版本）+ **App×品牌覆盖矩阵热力图**。
- **实现逻辑**：覆盖率 = 已自动化用例 ÷ 总用例；趋势按版本快照；热力图每格 = 该 App 在该品牌上的覆盖率%。**指标后续再调整**（用户暂未定稿）。

### 3.8 设备管理（devices）
- **需求**：品牌台架卡片（品牌 / udid / 分辨率 / 系统 / 在线状态 / 设备代理就绪），可刷新。
- **实现逻辑**：设备注册表 + 定时探活（adb devices + uiautomator2 atx-agent 健康检查）。

### 3.9 应用管理（apps）
- **需求**：被测 App 登记（App名 / 包名 / 主Activity / 版本 / 测试账号 / 关联用例数），可接入新 App。
- **实现逻辑**：App 注册表，供执行时 `am start` 与账号注入使用。

### 3.10 定时调度（sched）
- **需求**：定时回归计划（夜间全量 / 构建后冒烟 / 周末 Monkey 等），cron 表达式 + 范围 + 启停开关 + 下次执行时间。
- **实现逻辑**：APScheduler / 系统 cron 触发执行引擎；启停即开关任务。

### 3.11 系统设置（settings）
- **需求**：失败通知（企业微信/钉钉 Webhook + 推送策略）+ 成员与角色管理。
- **实现逻辑**：配置表 + 通知服务；成员表 + RBAC。

## 六、实现演进（相对原始需求，2026-07 记录）

原始需求以「一套框架 + `@pytest.mark.case` 标记」为前提。实现中出现了**多套并行的自动化框架**（团队用不同技术路线各写了一套），平台据此演进出以下能力，核心目标（一次写、多品牌跑、自动出报告 + 覆盖率）不变：

1. **手工用例库（分母）来源升级**：从「Excel 模板导入」升级为**导入官方测试报告 Excel**（One Info MEDIA Test Report，一 App 一 sheet，约 1256 条 P1~P4），作为真实分母；原模板导入保留。
2. **平台兼容多套框架（适配器架构）**：不强推平台自己的标记约定，每套框架写一个「适配器」把它的**用例↔官方编号映射**与**执行**接进来。已接三套：我们的 framework（case 标记，u2）/ Media_automation（ADB-XML dump，CSV 映射）/ Media_automation_Zcode（u2，markdown 覆盖矩阵映射）。上传 zip 自动识别框架。
3. **真机执行落地**：执行引擎策略式 —— 台架可达则真机跑对应框架的 pytest（实时进度/Allure/落库），否则回退模拟（保持原型节奏）。两个入口：按任务、按脚本。
4. **Allure 真报告**：真机产出 + 模拟/历史合成；执行完预生成，点击秒开。
5. **框架选型结论**：**Zcode（u2 元素级）为长期主力**（选择器分辨率无关＝跨品牌可行，且比 ADB-XML 坐标点击快 5 倍+）；Media_automation（ADB-XML）作稳定性兜底与车机技巧来源。
> 详细技术决策见 `项目文档.md §五`，跨会话续接见 `上下文.md`。
